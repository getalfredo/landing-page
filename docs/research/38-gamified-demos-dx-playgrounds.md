# Research: gamified demos and DX playgrounds (wayfinder #38)

Date: 2026-07-16. Method: three parallel primary-source passes (playable landing demos; scoring and leaderboard mechanics; with-vs-without framings), each against the live pages and official docs, followed by direct verification fetches for every load-bearing claim the first passes could not reach. The question being fed: a bigger sibling of the hero demo that lets a visitor play their way through what Alfredo does, possibly with scoring, possibly with a with-vs-without contrast mechanic. Judgments are grounded in the locked HQ identity, the honesty rule (SIMULATED FEED etch on anything that looks like product output, no fake numbers presented as real), and the built hero demo (`src/components/landing/hero-demo.tsx`: click-driven guided cycle Deploy -> Show dashboard -> + New project, reduced-motion fallback, plain React + CSS).

Fetch caveats, disclosed inline where they matter: keybr.com, regexcrossword.com and the Better-T-Stack builder at `/new` are JS-rendered and returned shells, so their runtime behaviour is described only where a second source owns it. The NYT Wordle help page is unreachable from this environment (direct and via archive), so Wordle storage claims are labelled unverified. Railway serves an agent briefing to non-browser fetchers, as already noted in #36. Mobile behaviour generally cannot be verified by fetching; no mobile claims are made beyond what pages state themselves.

## Part 1 - Gamified and playable demos: what actually works on the live pages

### The game is the page (zero-friction start)

- **monkeytype** (https://monkeytype.com): the fetch returned a JS app shell, but the shell itself carries the pattern: "Click here or press any key to focus", the test UI (wpm, accuracy, raw, consistency), a daily leaderboard link, and "Sign in to save your result". Positioning line: "A minimalistic, customizable typing test". No signup wall; the account exists only to persist and to qualify for leaderboards. The whole product is playable in the first second.
- **keybr** (https://www.keybr.com): JS-only; the fetch returned essentially the word "Practice". Its `/high-scores` page exists but rendered only a "High Scores" heading, so the table itself is unverified. Community threads on the official support group (https://groups.google.com/g/keybr) indicate account-gated progress persistence; anonymous fallback storage is unverified.
- The lesson: these pages convert because the first interaction costs nothing and produces a number about you. Neither puts the leaderboard first; the personal metric is the hook, the leaderboard is a retention feature behind an account.

### Real syntax as the game surface

- **Flexbox Froggy** (https://flexboxfroggy.com, fetched): "A game for learning CSS flexbox." You type actual CSS property/value pairs into a code field and real flexbox layout moves the frogs. The fetch showed a level selector, a Next control, language/difficulty/colorblind settings and links to sibling games. No login, no visible persistence mechanism in the fetched markup.
- **CSS Grid Garden** (https://cssgridgarden.com, fetched): same Codepip family, "A game for learning CSS grid layout", numbered level sequence, Reset and Next controls, 30+ languages.
- **Oh My Git!** (https://ohmygit.org, fetched): "An open source game about learning Git!" It embeds a real git instance ("execute arbitrary Git commands"), visualizes the repository graph in realtime, and layers playing cards over the commands as memory aids. Downloadable, not in-browser, which keeps its web page static.
- **Regex Crossword** (https://regexcrossword.com): JS shell, only the title rendered; mechanics not verified this pass.
- The lesson: the strongest teaching games do not simulate an abstraction of the tool, they run the real syntax and show its real consequences. For Alfredo the equivalent surface is the real CLI line the hero already shows (`alfredo up --with ...`) and the real HQ ledger, not an invented mini-game skin.

### Playground companies do not put the playground on the landing page

- **StackBlitz** (https://stackblitz.com, fetched): hero is "Turn text into working web apps" (Bolt-first now); the in-browser-dev claim is carried by copy ("WebContainers boots entire development environments in milliseconds, securely within your browser tab") and a comparison table, not by an embedded editor. The landing page itself is promotional, the playground lives behind the CTA.
- **CodeSandbox** (https://codesandbox.io, fetched): "Sandboxes built for scale", microVMs "spin up within 2 seconds"; the landing page embeds no live editor, and Sandpack (their embeddable editor) is a separate product in the nav.
- **Stripe** (https://docs.stripe.com/workbench, https://docs.stripe.com/workbench/shell): the interactive surface is Workbench with a Shell/API explorer, and docs personalize code samples with your own test keys once signed in. It is an authenticated evaluator surface, not a cold-visitor toy. (Search-verified against Stripe's own docs pages; the docs' unauthenticated rendering was not fetched this pass.) stripe.dev, sometimes remembered as a demo/game hub, is currently a plain engineering-blog index with no games or interactive demos (fetched https://stripe.dev).
- The lesson: even companies whose product is "instant playground" treat the landing page as a static argument with a fast door into the real thing. A waitlist page with no product access has even less reason to fake a full product playground; the demo has to be honest theater, which is what the hero already is.

### Big playful set pieces and where they are licensed

- **Bruno Simon** (https://bruno-simon.com, fetched): the portfolio is a drivable WebGL world (Three.js, Rapier physics, Howler audio, per its own Behind the Scene section), with an achievement system and copy that says "Please drive around to learn more about me and discover the many secrets of this world." It works because the genre is a personal portfolio and the gimmick is the resume. No no-JS fallback was observable in the fetch.
- **Figma** (https://www.figma.com, fetched): the current homepage is conventional; hero "The intelligent canvas for infinite creativity", product mockups, template carousels. No live canvas or cursor toy was visible in the markup. The remembered "cursor party" homepage is not there today; the multiplayer-cursor tech is documented as an in-product feature (https://help.figma.com/hc/en-us/articles/4403130802199-Use-cursor-chat-in-Figma-Design).
- **Vercel and Linear**: surveyed in #36 (`docs/research/36-stunning-landing-pages.md`); neither carries a playable element, both argue with product frames and copy. Not re-fetched.
- The lesson: full-page games are licensed by genre (portfolio, education). Dev-infra pages that tried them have retreated to static argument plus one contained interactive element, which is the slot the hero demo already occupies. A bigger sibling should stay a contained section, not take over the page.

### Guided tours, and the skimmability numbers

- **Arcade** (https://www.arcade.software, fetched): hero "Your product story, built by AI in minutes." The homepage does not visibly embed a live Arcade tour; it shows static showcases with tab switching. The product is captured-screen click-through demos plus AI video.
- **Navattic** (https://www.navattic.com, fetched): "Accelerate the buyer journey with demos"; no embedded self-demo was verifiable in the fetched markup. Their State of the Interactive Product Demo 2026 report (https://www.navattic.com/report/state-of-the-interactive-product-demo-2026, their own data across 40,000+ demos) is the useful primary artifact: top-percentile demos reach 84.4% engagement past step one and 61.6% completion; recommended shape is 5-13 steps per flow with dialog text held to 25-30 words.
- The lesson: 5-13 steps and 25-30 words per beat is the published envelope for an interactive walkthrough that people finish. The hero demo's three-beat cycle sits comfortably inside it; a bigger sibling should stay under roughly a dozen deliberate clicks.

### Load weight, mobile, reduced motion

- Load tactics observed: click-to-start is universal (monkeytype focuses on click/keypress; the hero demo already advances only on deliberate clicks). Oh My Git keeps the heavy thing out of the browser entirely. StackBlitz/CodeSandbox keep the landing static and defer the heavy runtime behind a CTA. Bruno Simon front-loads a loading screen because the game is the site; that cost is not available to a landing page with a 10-second leave hazard (#37).
- Reduced motion: the canonical guidance is the `prefers-reduced-motion` media query (https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion, https://web.dev/learn/accessibility/motion): disable or collapse non-essential motion for users who set it. The repo already has `use-prefers-reduced-motion.ts` and the hero pre-fills its form for reduced-motion visitors; any bigger game must ship a same-grade static fallback, not a stripped-down animation.
- Mobile: none of the fetched pages document their mobile game behaviour, and fetching cannot verify it. The known local fact is that the hero Deploy flow already has a mobile dead-end owned by #26/#47; a bigger interactive sibling multiplies that question and should not be scoped without a mobile answer.

## Part 2 - Scoring, high scores, leaderboards

### What the precedents actually show

- **monkeytype's leaderboard is expensive on purpose.** Entry requires an account with roughly two hours of typing history (maintainer/community threads on the official repo: https://github.com/monkeytypegame/monkeytype/discussions/5307, https://github.com/monkeytypegame/monkeytype/discussions/1460). The public repo's anti-cheat module is a deliberate stub: `backend/src/anticheat/index.ts` (https://github.com/monkeytypegame/monkeytype/blob/master/backend/src/anticheat/index.ts) exposes `validateResult`/`validateKeys` signatures built around per-key timing statistics but logs "No anticheat module found, result will not be validated" unless a private module is wired in. A purpose-built typing product with a backend team keeps its score validation closed-source and gates the board behind accounts. That is the real cost of a public leaderboard done properly.
- **Wordle's mechanic is the opposite and it scaled further.** Personal stats and streaks per browser, no server leaderboard, and the share artifact is a copy-paste emoji grid. Caveat: the NYT help page (https://help.nytimes.com/hc/en-us/articles/4406941603092-Wordle) could not be fetched from this environment, direct or via archive, so the storage detail is common knowledge, not fetch-verified here; the widely reported streak-loss-on-cleared-storage complaints are consistent with client-side persistence.
- **keybr** runs a high-scores page (https://www.keybr.com/high-scores, exists but JS-rendered, table unverified) with account-backed persistence, again pairing a board with accounts.
- **Cheap plausibility checking has a citable pattern.** OWASP's input validation guidance states the base fact: client-side validation is UX only, anything client-submitted is forgeable (https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html). OWASP Juice Shop documents a real lightweight cheat detector: it compares elapsed time between solves against a minimum plausible time per difficulty (https://help.owasp-juice.shop/appendix/cheat-detection.html). Minimum-plausible-duration checks are the affordable tier below replay validation.
- No dev-tool marketing page with a live public high-score table was verified this pass. The pattern that has verified precedent on marketing-adjacent surfaces is the personal metric plus share artifact, not the global board.

### Mapped onto the existing stack

- **Ephemeral in-memory**: score lives in component state, gone on reload. Zero build cost beyond the game itself, zero abuse surface. This is what the hero demo implicitly does today (no score).
- **localStorage personal best**: the Wordle model. A few lines of code, no backend, no moderation. A visitor can only cheat themselves. Survives revisits on the same browser, which is exactly the return-visit hook a waitlist page wants.
- **PostHog event carrying the score**: `posthog.capture()` takes an arbitrary properties map (https://posthog.com/docs/product-analytics/capture-events); cookieless mode is documented (https://posthog.com/docs/privacy/data-collection); the capture API's only documented payload limit is 20MB per batch request (https://posthog.com/docs/api/capture). A `demo_completed` event with `{steps, seconds, score}` costs no new endpoint and no schema work. Scores remain client-submitted and forgeable, but nothing is displayed back publicly, so forgery pollutes analytics at worst. Note the current event budget is exactly four events; adding demo events is a deliberate expansion to record in the analytics decision (#21 lineage).
- **sqlite aggregate counter (no names)**: a drizzle table plus one endpoint reusing the waitlist's honeypot and rate-limit pattern (`email/created_at/consent_text/source` table already demonstrates the shape). Storing only anonymous aggregates (runs completed, median steps) keeps zero moderation burden and produces a number honest enough to earn the LIVE etch that #50 already owns. Abuse surface: endpoint spam inflating counters; the existing rate limit plus a Juice-Shop-style minimum-duration check covers the cheap attacks.
- **Server-backed named leaderboard**: needs a write endpoint, a display surface, forgery resistance (monkeytype-grade: accounts, private validation) and free-text name moderation. Alfredo has no accounts and no moderator. Every verified precedent that runs one pays costs this project cannot pay pre-launch.

## Part 3 - With-vs-without and effort-comparison framings

- **Better-T-Stack** (https://better-t-stack.dev, https://better-t-stack.dev/analytics): the ticket suspected a time-saved estimate; there is none, anywhere on the site. The homepage carries the generated command (`bun create better-t-stack@latest`) and links the builder at `/new` (JS-rendered, picker not verifiable by fetch). Its honest artifact is the `/analytics` page: a public telemetry dashboard showing 52,361 tracked project starts, an explicit disclosure that this undercounts real usage (estimated ~111.4K all-time), a per-day average and stack-choice frequency charts. Pure adoption counting, no productivity spin.
- **create-t3-app** (https://create.t3.gg): runs no effort comparison at all. Its stance is curation: "Streamline the setup of typesafe Next.js apps WITHOUT compromising modularity", "This is NOT an all-inclusive template", "Take what you want and nothing more." A counter-example: the category leader for setup scaffolding does not dramatise the setup tax, it understates itself.
- **Railway** (https://railway.com, https://docs.railway.com/platform/compare-to-heroku): the fetch received the agent briefing ("This document is your setup briefing"); its whole comparison argument is one command, `railway up -y`, described as handling login, project creation, service creation, deploy and URL in one loop. The human-facing Heroku comparison doc juxtaposes raw public prices (Heroku tiers vs $20/vCPU-month) and leaves the arithmetic to the reader; no synthesized "you save $X" figure.
- **Render** (https://render.com/docs/render-vs-heroku-comparison): the instructive mixed case. One verifiable anchor ("a web service on Render with 2GB of RAM is $25/mo. A Heroku dyno with 2.5GB of RAM is $250/mo", both prices publicly checkable) sits next to an undisclosed-methodology headline ("customers... often see cost reductions of over 50%... saving thousands of dollars every month"). The two are not even reconciled: the worked example implies ~90%, the headline says 50%+. This is what inflated framing looks like from the inside.
- **Bun** (https://bun.sh): the honesty benchmark, as in #36. Every multiplier names the hardware (Linux x64, Hetzner) and links a public benchmark repo (github.com/oven-sh/bun/tree/main/bench/install, github.com/rolldown/benchmarks). Claims are reproducible, not asserted.
- **Coolify** (https://coolify.io): names the alternatives ("alternative to Vercel, Heroku, Netlify and Railway") and quantifies nothing. Positioning without measurement, which is honest by omission.
- **Side-by-side terminal race**: no live, verifiable example of an animated with-tool vs without-tool race was found on any dev-tool page this pass. If Alfredo builds one it is not copying a proven pattern, it is originating one, and it must be deterministic and etched as simulation.

Honesty ranking that falls out of the set: countable steps the visitor can verify on screen > measured time on named hardware with a public repro > raw price/step juxtaposition with the math left to the reader > transparent usage telemetry that discloses its own gaps > estimated savings with a disclosed formula > undisclosed aggregate percentages and dollar figures. Alfredo's honesty rule caps it at the tiers where every number is either the visitor's own action count or a scripted count visibly derived from a script shown on screen.

## Part 4 - Mapping to the HQ identity and the honesty rule

- Game-is-the-page (monkeytype): translates as posture, not scale. Zero-friction, press-a-key-and-something-real-happens is already console-native. Taking over the page clashes with the narrative spine.
- Real-syntax surface (Froggy, Oh My Git): translates directly. The playable thing should be the real CLI line and the real HQ ledger, both already in the demo vocabulary. Inventing a fictional mini-game skin (frogs, gardens) clashes with the dark-glass material and the plain-English rule.
- Playground-behind-the-CTA (StackBlitz, CodeSandbox): translates as restraint. The landing page stays an argument; the interactive element stays contained and click-mounted, never autoplaying weight.
- Full-page WebGL world (Bruno Simon): clashes. Wrong genre license, wrong load budget, wrong material.
- Guided tour shape (Navattic's 5-13 steps, 25-30 words): translates as a hard envelope for any bigger sibling.
- Personal best + share artifact (Wordle): translates cleanly. A visitor's own click count or elapsed time is a real number, needs no SIMULATED etch, and localStorage persistence has no abuse surface.
- Global named leaderboard (monkeytype): clashes. Requires accounts and private anti-cheat to be non-fake; a forged or unmoderated board violates the no-fake-numbers rule the moment it renders.
- Anonymous real aggregate ("N runs played"): translates, and can carry the LIVE etch from #50, because it is a true number produced by this page.
- Deterministic with-vs-without race: translates if and only if both lanes are scripted, the step counts come from a script the visitor can read, and the whole panel carries the SIMULATED etch. A race against a named competitor (Better-T-Stack) is now also factually awkward: their site makes no time claim to race against, so the honest opponent is "doing it by hand", enumerated step by step.
- Undisclosed savings math (Render's headline tier): clashes outright. No dollars, no invented hours.

## Shortlist for the grilling ticket (#40)

Options to react to, not a single recommendation. Build weights are relative to the existing hero demo (React + CSS, no canvas).

1. **The Setup Race.** Split console: left lane scripts the by-hand wiring of auth/email/db/analytics/secrets as a ticking step counter, right lane is the visitor clicking through `alfredo up`. Score is steps skipped, derived from the visible script. Scoring: step count only. Persistence: none (ephemeral). Honesty: both lanes SIMULATED etched; every counted step is on screen and enumerable. Build: medium; it is a second hero-demo-class component plus a copy ticket for the manual-lane script.
2. **Tray Panel Speedrun.** A timed bigger sibling of the hero checklist: wire five trays for a new project by keycap clicks, clock and click count running. Scoring: elapsed seconds and clicks, personal best. Persistence: localStorage, optional anonymous PostHog event. Honesty: the time and clicks are the visitor's real actions, no etch needed on the score itself; the deploy it triggers stays SIMULATED etched. Build: small-medium; heaviest reuse of existing demo vocabulary.
3. **Ship Week.** An operations game: run a handful of simulated days across three projects from the HQ ledger; events arrive (bounce spike, failed job, cert expiry) and wired trays resolve each in one click; the close screen shows the scripted count of manual interventions the same week costs without an HQ. Scoring: interventions avoided, scripted count. Persistence: localStorage personal best. Honesty: SIMULATED FEED throughout; the without-HQ tally must be a step count from a visible script, never invented hours. Build: heavy; new game logic, new copy, the only concept that sells the operations premise rather than deployments.
4. **Type the Line.** A monkeytype-shaped micro-moment: type the real `alfredo up --with auth,email,db,analytics` line, get your seconds, and see it next to the enumerated manual checklist those flags replace. Scoring: seconds, personal metric only. Persistence: ephemeral or localStorage. Honesty: the typed line is the real CLI syntax, the checklist is countable; nothing simulated except the etched response output. Build: small; the cheapest way to test whether play mechanics move waitlist conversion at all.
5. **Real aggregate strip (companion, not a game).** Whatever concept wins, an anonymous server-backed counter ("runs completed: N", "median steps skipped: M") via a drizzle table reusing the waitlist honeypot/rate-limit pattern, surfaced with the LIVE etch. Scoring: aggregate only, no names. Persistence: sqlite. Honesty: a true number produced by this page; coordinates with #50. Build: small server work plus one etch line.

Explicitly not shortlisted: a named public high-score board. Every verified precedent (monkeytype, keybr) pays for one with accounts, private anti-cheat and moderation; without those it becomes a fake-number surface, which the honesty rule forbids. It stays available as a post-launch idea if accounts ever exist.

## Notes and surprises for follow-up

- Better-T-Stack has no time-saved claim; the assumption baked into this ticket's framing was wrong, and its `/analytics` transparency page is the pattern worth borrowing (real counts that disclose their own undercounting).
- stripe.dev currently hosts no games or demos; it is a blog index. Any memory of Stripe demo games refers to surfaces that are not live today.
- No live dev-tool side-by-side race exists to copy. Concept 1 would be an original pattern; that is an argument both for it (distinctive) and against it (no evidence it converts).
- Navattic's 2026 report figures (84.4% engagement, 61.6% completion for top demos, 5-13 steps, 25-30 words) are the only published envelope numbers for interactive-demo length found on a primary source; use them as the scope ceiling in #40.
- The keybr high-scores table, the Better-T-Stack builder UI and the regexcrossword mechanics remain unverified (JS-only pages); none is load-bearing for the shortlist.
- Any concept chosen must land a mobile answer alongside #26/#47's existing hero mobile dead-end, and a reduced-motion fallback of the same grade as the hero's pre-filled form.
