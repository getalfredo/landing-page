## Part 1 — What the sources actually say

### SEO for a single marketing page

- Google generates title links from `<title>`, H1, `og:title` and more, and rewrites freely; no official character limit — truncation is by pixel width, "60 chars" is [folklore]. Brand name at start or end with a delimiter. [vendor] https://developers.google.com/search/docs/appearance/title-link
- Snippets come primarily from page content; the meta description is used only when it's better. No official length limit. [vendor] https://developers.google.com/search/docs/appearance/snippet
- Canonical is not required, but Google endorses self-referencing canonicals — cheap insurance against `?utm_*`, host and slash duplicates from launch links. [vendor] https://developers.google.com/search/docs/crawling-indexing/consolidate-duplicate-urls
- Heading count and order don't matter for ranking ("you can use H1 tags as often as you want" — John Mueller; starter guide: order matters for accessibility, not Search). Landmarks are accessibility, not ranking; "landmarks boost SEO" is [folklore]. [vendor] https://developers.google.com/search/docs/fundamentals/seo-starter-guide
- Core Web Vitals thresholds at p75: LCP ≤ 2.5 s, INP ≤ 200 ms, CLS ≤ 0.1. Ranking weight per Google's own wording is a tiebreaker ("no single signal… can contribute to success"), not a primary factor. [vendor] https://web.dev/articles/vitals, https://developers.google.com/search/docs/appearance/page-experience
- Fonts (all [vendor] https://web.dev/articles/font-best-practices): WOFF2 only; preload only the 1–2 critical files; `font-display: optional` is most performant, `swap` acceptable for a branded display face; `size-adjust`/overrides make the fallback swap near shift-free (https://web.dev/articles/css-size-adjust). Self-hosting wins only when served well.
- Transform/opacity-only animations are CLS-safe by construction; never animate layout properties. [vendor] https://web.dev/articles/optimize-cls
- A `<video>`'s poster is an LCP candidate — treat it like a hero image. [vendor] https://web.dev/articles/lcp
- robots.txt is optional (404 = no restrictions; 5xx is harmful). A sitemap is unnecessary by Google's own criteria (≤~500 pages, well linked) — a one-page site trivially qualifies. [vendor] https://developers.google.com/search/docs/crawling-indexing/sitemaps/overview
- Structured data has documented value beyond rich results: "Google uses structured data… to understand the content of the page, as well as to gather information about the web and the world in general." JSON-LD is the recommended format. [vendor] https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data
- **FAQPage rich results are dead for everyone**: restricted to government/health sites Aug 2023, deprecation notice May 8 2025, removed from Google Search entirely as of May 7 2026 (docs pulled June 15 2026). [vendor] https://developers.google.com/search/updates
- `WebSite` schema is "most important" among the signals for the site name shown in SERPs; `Organization` schema disambiguates the entity and feeds logo/knowledge panel. [vendor] https://developers.google.com/search/docs/appearance/site-names, https://developers.google.com/search/docs/appearance/structured-data/organization
- `SoftwareApplication` rich results require `offers.price` + ratings — a pre-launch product cannot honestly qualify; a minimal node for entity understanding only is fine. [vendor] https://developers.google.com/search/docs/appearance/structured-data/software-app
- OGP requires `og:title`, `og:type`, `og:image`, `og:url`; `og:site_name` and image dimensions optional. Meta's image spec: 1200×630 for high-res, ~1.91:1. X falls back to `og:*` when `twitter:*` tags are absent, so `twitter:card` is the only strictly needed twitter tag ([vendor], X's primary docs currently unreachable — figures via mirrors). [spec] https://ogp.me/, [vendor] https://developers.facebook.com/docs/sharing/webmasters/images/

### AEO / GEO (being cited by AI answer engines)

- Google's official position: AI Overviews / AI Mode need nothing special — "no additional requirements… no special schema… no AI text files." Eligibility = indexed and snippet-eligible. [vendor] https://developers.google.com/search/docs/appearance/ai-features
- ChatGPT search: allowing `OAI-SearchBot` in robots.txt is OpenAI's one documented requirement — "sites that are opted out… will not be shown in ChatGPT search answers." Bing is the only named third-party provider, so normal SEO **including Bing** (Webmaster Tools, IndexNow) is the entry ticket. [vendor] https://developers.openai.com/api/docs/bots
- Perplexity: `PerplexityBot` surfaces sites in results, is not used for training; no first-party citation-ranking documentation exists — anything specific in SEO blogs is inference. [vendor] https://docs.perplexity.ai/guides/bots
- Vendors split training bots from search bots: you can block training (`GPTBot`, `ClaudeBot`, `Google-Extended`) while keeping answer-engine visibility (`OAI-SearchBot`, `PerplexityBot`, `Claude-SearchBot`, Googlebot). For a page that wants LLMs to know the product, allowing everything is the visibility-maximizing default. [vendor] https://support.claude.com/en/articles/8896518, https://developers.google.com/search/docs/crawling-indexing/google-common-crawlers
- Infrastructure gotcha: Cloudflare blocks AI crawlers **by default** on new zones since July 1 2025. If the deploy path adds Cloudflare or similar, the page can be invisible to answer engines with a permissive robots.txt. [vendor] https://blog.cloudflare.com/content-independence-day-no-ai-crawl-without-compensation/
- The only controlled experiment (GEO paper, Aggarwal et al., KDD 2024): quotation addition +41%, statistics +33%, cited sources +28% answer visibility; keyword stuffing ≈ nothing; low-authority pages gained most — exactly what a new landing page is. [research] https://arxiv.org/abs/2311.09735
- llms.txt ([spec] https://llmstxt.org/): markdown briefing at the root — H1 name, blockquote summary, H2 link lists. Real adoption verified by fetch (Anthropic, Cloudflare, Vercel, Fly.io at `/llms.txt`, Railway) — but **no AI vendor has committed to reading it**; Google staff compare it to the keywords meta tag. Honest framing: a docs-for-coding-agents convention, zero-risk, unproven for AEO. Relevant anyway: Alfredo's audience runs agents (flagged in #36's findings).
- Q&A framing, extractable definitions, consistent entity naming: mechanistically sensible and loosely consistent with the GEO findings, but [folklore] in production engines — no vendor documents them.
- Citation-vs-ranking overlap studies disagree by 2–4× (Ahrefs 76%→38%→12%, Semrush ~20–26%) — ranking helps, the mapping is loose, don't expect measurability. [secondary]
- Biggest lever is off-page: a lone landing page with no external corroboration (GitHub README, HN/Reddit threads, "Alfredo vs Coolify" mentions) rarely gets cited regardless of on-page work. [inference, flagged]

### Conversion for a waitlist-capture page

- Leave-hazard peaks in the first ~10 seconds; a clear value proposition is what holds survivors (Weibull analysis of ~205k visits). 57% of viewing time is above the fold, 74% in the first two screenfuls. [research] https://www.nngroup.com/articles/how-long-do-users-stay-on-web-pages/, https://www.nngroup.com/articles/scrolling-and-attention/
- First impressions form in ~50 ms and halo into perceived credibility — for a zero-user product, visual polish IS the trust signal. [research] https://www.tandfonline.com/doi/abs/10.1080/01449290500330448
- Stanford Web Credibility (Fogg, ~4,500 participants): show the real people behind the site, make claims verifiable, professional design, easy contact — directly supports the named founder note and the open-source commitment. [research] https://credibility.stanford.edu/guidelines/
- Discovered fakery leaves users worse off than no signals at all; no fake counts, no fake testimonials. [research, mixed quality]
- CTA: button labels should state the exact outcome (never "Submit"/"Get Started") [research NN/g]; repeating the CTA down a long page and hero-form-vs-scroll are [case study/folklore] — direction plausible, unquantified.
- Form mechanics (Baymard, [research]): single field single column; visible persistent label, not placeholder-only; validate on blur, never while typing; clear the error live on correction; positive green-check validation reduced anxiety; error messages specific and adjacent. https://baymard.com/blog/inline-form-validation
- Success state should confirm AND set expectations (what happens next) — NN/g visibility of system status. [research]
- Waitlist referral loops: Harry's is the only case with first-party numbers (~100k emails/week, 77% via referral, open-sourced app). [case study] https://tim.blog/2014/07/21/harrys-prelaunchr-email/ — queue-position gamification itself has no controlled study.
- FAQ sections: real objection-handling value, no measured conversion effect; placement before the final CTA is [folklore] but consistent with the objection-handling rationale. Bonus: FAQ content doubles as the quotable/extractable material the GEO evidence favors.

## Part 2 — Where the built page stands

The page is in good shape on the fundamentals. Full SSR (nitro node server): a no-JS crawler receives every heading, the FAQ as a semantic `<dl>` with answers always in the DOM, the founder note and the form markup (`vite.config.ts:16-18`, verified in the server bundle). Head has locked title/description, self-referencing canonical, OG tags with a purpose-made 1200×630 image, `twitter:card` + `twitter:creator` (`src/routes/__root.tsx:11-61`). One H1 with an SR-static fallback for the cycling word; clean h2 tier; `header/main/footer/section` landmarks; act videos carry posters, labels and reduced-motion stills. robots.txt is allow-all. Copy already leads with an extractable definition ("Alfredo is the home for your projects…") and consistent entity naming through hero, FAQ, footer microprint and meta description. The form is one email field with a visible slot label, honeypot, rate limit, idempotent duplicates, and a confirmation that reports a real queue position. CTA copy ("Join the waitlist") is specific and outcome-oriented; FAQ sits before the final CTA per narrative v2.

Gaps found (detail in Part 3): no structured data of any kind (deliberate #21 call, now stale); OG/Twitter set incomplete (`og:site_name`, `og:image:width/height/alt` missing); `/privacy` inherits the root canonical and title (`src/routes/privacy.tsx:12-14` has no head override); no llms.txt; the build ships 29 woff2 files across all script subsets (~460K of non-latin waste — #33's "five files, acceptable" ruling counted wrong) with no preload; 518K main JS chunk; the waitlist form is fetch-only with no `action` fallback (`waitlist-form.tsx:52`), validates natively only, and the error state is the non-specific `SIGNAL LOST · TRY AGAIN`; success state doesn't set expectations beyond the consent line; manifest.json exists but is unlinked (owned by #48).

Already owned elsewhere — not re-proposed here: #48 owns the manifest link + theme-color, PostHog placeholder guard, devtools prod gate, signup-event inflation, act-video `autoPlay`→`preload="metadata"` (~1.8MB), 390px header crowding, drizzle.svg. #24 (Go live) owns the Lighthouse/CWV measurement. #26/#47 own the mobile Deploy dead-end. #50 owns the real-signal LIVE etch (real waitlist count named as a source). #54/#56 own concrete-number copy — which is also the GEO paper's highest-lift move, so those tickets carry AEO weight beyond their visual intent.

## Part 3 — Prioritised gap list

P1 — clear evidence, cheap, do before launch:

1. **JSON-LD: partially reverse #21's skip.** Add `WebSite` (drives the SERP site name — Google calls it the most important signal for that) and `Organization` (entity disambiguation, logo ≥112×112, `sameAs` → GitHub org + @alperortac); optionally a minimal `SoftwareApplication` node with no ratings/offers. Explicitly do NOT add `FAQPage` — the rich result is removed for all sites as of May 2026 — and do not fake rating eligibility. → new ticket.
2. **Head completion + per-route heads.** `og:site_name`, `og:image:width/height/alt`; give `/privacy` its own title, description and canonical (it currently claims `https://getalfredo.com` — a cross-page canonical collision). Verify twitter fallback needs nothing beyond `twitter:card`/`twitter:creator`. → new ticket.
3. **Font loading pass.** Subset to latin/latin-ext (drops ~460K), preload the 1–2 files used above the fold, keep `swap` for the display faces but tune fallbacks with `size-adjust`, consider `optional` for body Inter. Reopens #33's font ruling, which under-counted the shipped files. → new ticket.

P2 — evidence-backed, small, worth doing:

4. **Waitlist form friction pass.** Add a no-JS path (form `action` posting to the API as progressive enhancement), validate on blur with live error clearing and a positive valid-state check, make the error message specific (network vs invalid vs rate-limited — `SIGNAL LOST` covers all three today), and let the success state set expectations (Baymard/NN/g backed; keep the Console voice). → new ticket.
5. **llms.txt agent briefing.** One markdown file per the llmstxt.org shape (name, blockquote summary, links to the page and privacy note, the FAQ's canonical integration list). Zero AEO evidence, but the audience runs coding agents, adoption among peer infra products is real (Fly.io, Railway, Vercel), and it closes #36's open pointer. → new ticket.
6. **Answer-engine access at go-live.** Keep robots.txt permissive (explicitly allow `OAI-SearchBot`, `PerplexityBot`, `Claude-SearchBot`; decide the training-bot stance — default allow for visibility); verify no CDN/proxy layer blocks AI crawlers (Cloudflare blocks by default since July 2025); submit to Bing Webmaster Tools + IndexNow (Bing feeds ChatGPT search); confirm Search Console indexing. Only meaningful against the live domain. → new ticket, blocked by #24.

No action, decided by this research:

- **Sitemap.xml: skip.** A one-page site meets Google's own skip criteria; adding one is cargo cult.
- **Heading/landmark rework: none needed.** Count and order are ranking-irrelevant; current structure is clean for accessibility.
- **CWV panic: none.** Thresholds are a tiebreaker; measurement stays with #24's Lighthouse pass. The 518K main chunk is noted; revisit only if the p75 numbers say so.
- **Fake-proof anything: never.** Testimonials/logo walls stay rejected (#36); the honest set (founder note, open source, real queue position, future LIVE etch) is exactly what the credibility research supports.
- **Referral loop: out of this ticket.** The only evidenced waitlist growth mechanic (Harry's, 77% via referral), but it's a feature decision against #4's locked single-opt-in mechanics, not a fix. Recorded here as a hypothesis for a future effort; instrument first, decide later.
